# Student Assignment

Use this reference when the user asks for a school-style use case diagram matching the assignment described in this workspace.

## Required minimums

- Create one use case diagram.
- Include at least six use cases.
- Include at least one `include` relation.
- Include at least one `extend` relation.
- Include at least one inheritance/generalization relation.

## Practical interpretation

- Treat the list above as a minimum, not a target ceiling.
- Keep every required relation semantically justified.
- If the chosen system does not naturally support one required relation, adjust the scenario instead of inserting an invalid UML relation.
- Prefer a domain that naturally yields at least two actor roles, one reusable mandatory sub-flow, one optional or conditional sub-flow, and one clear specialization.
- Keep the final diagram compact. A larger system can still be modeled with fewer than ten use cases if the chosen goals are at the right abstraction level.

## Delivery preference

- Prefer readability over completeness.
- Keep the number of bubbles and arrows as low as possible while preserving the required semantics.
- Group related use cases so that relation arrows stay short.
- Use actor generalization when it removes duplicated actor-to-use-case links without distorting the meaning.

## Suggested delivery

- Brief actor list
- Brief use-case list
- Short rationale for each non-trivial relation
- Editable diagram source, preferably PlantUML unless the user requests another format
